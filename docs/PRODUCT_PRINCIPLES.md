# Product Principles

This document complements [HIGH_LEVEL_ARCHITECTURE.md](./HIGH_LEVEL_ARCHITECTURE.md).

> **Purpose**
>
> This document defines the product philosophy that guides every product, UX and architectural decision.
>
> Whenever multiple valid solutions exist, these principles take precedence over implementation convenience.

---

## 1. The Hub is the user's digital home

The product should feel like the place where users begin their digital day. The hub is the trusted entry point to their digital life.

## 2. The product removes unnecessary digital effort

Every feature should make it easier, faster, and safer to reach digital services. The product should remove unnecessary steps, not add new ones.

## 3. The product never replaces the services it connects to

Banks, government, healthcare, and commercial websites remain the systems of record. The hub organizes access to them.

## 4. Security should be visible and reassuring

Security is not only technical.

Security should never surprise the user.

Users should always understand when the vault is locked, unlocked, or about to use sensitive information.

## 5. Advanced capabilities must never complicate the default experience

The default experience should remain extremely simple. Advanced capabilities should be additive rather than intrusive.

## 6. The user always remains in control

> **The product suggests. The user decides.**

The system assists users without taking control away from them.

The user always remains responsible for authentication and sensitive actions.

## 7. Build for trust before convenience

When trust and convenience conflict, trust wins. The product should always favor user confidence over automation.

## 8. Learn from users before expanding the model

Core product concepts should evolve from validated user behavior rather than assumptions. The product intentionally leaves room for learning.

## 9. Simplicity is a feature

Every new capability should preserve the feeling that the product is simple.

Complexity may exist internally, but should never become the default user experience.

The default citizen experience always has priority over advanced workflows.

---

## Document Stability

**Version 1.0**

These principles are intentionally stable.

Features, user interface, implementation details and architecture may evolve over time.

These principles should change only when the overall product vision changes.

They are intended to guide long-term product decisions rather than short-term implementation choices.
